## 一、项目概况

### 1.1 建设目标

为广州生产力促进中心建设 **中试服务全流程数字化管理系统**，实现：

- **Web 分角色门户**：调度员、审核员（中心管理端）；企业、技术人员（业务门户）
- **微信小程序移动补充**：企业端 16 页 + 技术人员端 6 页（不含调度员/审核员）
- **五模块业务闭环**：需求 → 评估 → 调度 → 反馈 → 档案 → 结案（`CLOSED`）

### 1.2 技术架构（已定稿，禁止随意更换）

| 层次 | 技术 | 仓库 |
|------|------|------|
| 后端 | Spring Boot 3 + MyBatis-Plus + JWT + Knife4j | `server/` |
| Web | Vue 3 + Vite + Element Plus + Pinia | `web/` |
| 小程序 | uni-app（Vue 3） | `miniapp/` |
| 数据库 | 开发默认 H2 文件库；论文/部署可切换 MySQL | `server/src/main/resources/db/` |
| 原型与规范 | 79 页 HTML/PNG + 业务活动图 | `原型图/`、`业务活动图/`、`指南规范/` |

### 1.3 四角色与端别分工（不可变更）

| 角色 | 端别 | 主要职责 |
|------|------|----------|
| 企业项目负责人 | Web + 小程序 | 需求提交、材料补充、签收、进度查看、反馈 |
| 中试技术人员 | Web + 小程序 | 任务接收、进度填报、结果提交/修改 |
| 中试调度员 | **仅 Web** | 受理、评估前置、匹配派发、督办、统计 |
| 中试审核员 | **仅 Web** | 受理材料审核（核验+受理决定）、可行性/结论审核、报告复核、档案结案 |

## 三、文档治理体系（防止偏离路线的依据链）

任何需求变更、增删页面、改流程，必须 **自上而下** 同步，禁止直接在代码里「顺手改」：

```

        ↓

        ↓

        ↓

        ↓

        ↓
```

**本文档（项目总体规划）** 位于最上层，负责 **阶段节奏、质量门禁、业务活动修订（专章如 §十七）、论文与答辩排期**；流程细节变更须 **先写入本文对应专章**，再按 §三 依据链向下游文档同步。

### 3.3 规划文档唯一性原则（强制）

| 规则 | 说明 |
|------|------|
| **唯一规划载体** | 阶段计划、走查结论、模块修订、答辩排期等 **全部整合进本文**；在本文中新增或扩展专章（如 §十七），并更新 §十六 修订记录 |
| **禁止另建** | **不得**在 `分工/` 或其他目录新建「××修订方案」「××规划补充」等独立 Markdown；已废弃的独立规划稿须删除，内容并入本文 |
| **允许并存的分工文档** | 仅 **`代码编写文档.md`**（编码施工图）、**`系统测试文档.md`**（测试验收）；二者承接本文决策，**不替代**本文做规划批准 |
| **修订写法** | 新修订用「§N 模块名 + 修订日期 + 已批准」专章；小改动在 §十六 修订记录一行说明并在相关章节改正文 |

### 3.4 功能变更前置阅读（强制）

| 规则 | 说明 |
|------|------|
| **先读本文** | 每次对系统功能进行 **增、删、改、查**（含页面、流程、接口、状态机、UI 交互）**必须先阅读本文** 及对应专章（如 §十七），确认范围、边界与已批准修订 |
| **先写规划、后写代码** | 变更意图须 **先写入本文**（新增/扩展专章或 §十六 修订记录），标注「已批准」或走查结论后，再按 §三 依据链同步下游文档，**最后**才进入 `server/`、`web/`、`miniapp/` 编码 |
| **禁止跳步** | **禁止** 未更新本文即在代码中「顺手改」流程或 UI；**禁止** 在 `分工/` 下另建独立规划稿替代本文（见 §3.3） |
| **编码角色自检** | 动手前对照：本文是否已有对应条款？`代码编写文档.md` / `页面清单.md` 是否已同步？测试用例是否需更新？ |
| **对话前必读** | 规划、走查、编码、测试相关 **每次对话开始前** 须通读本文（至少 §三、§十七 与 **§17.11 实现状态**），再提出或执行变更 |
| **用户明确「不用写代码」** | 当用户或规划角色 **明确指示仅更新规划、不写代码** 时，**只允许** 修改本文及 §三 依据链规定的下游文档；**严禁** 在同一轮次内改动 `server/`、`web/`、`miniapp/` 任何源码 |
| **未批准不得编码** | 本文 **无对应已批准条款** 的功能（如 §17.14 标记「已批准、未实现」），**不得** 提前在代码中实现；实现完成后须回填 §17.11 实现状态 |

### 3.5 规划助手（AI）工作边界（强制）

> **适用范围：** 本人通过 Cursor 等工具调用 AI 进行 **规划、走查、需求整理、文档修订** 时；与 §二「代码编写」角色 **严格分离**。

| 规则 | 说明 |
|------|------|
| **唯一产出物** | 规划助手在本项目中的 **唯一允许修改的仓库文件** 为 **`分工/项目总体规划.md`（本文）**；任务即 **只写规划、不写代码** |
| **禁止编码** | **严禁** 修改 `server/`、`web/`、`miniapp/` 及任何源码、配置、SQL、构建脚本；**严禁** 擅自改动 `代码编写文档.md`、`系统测试文档.md`（除非用户 **另行明确授权** 同步下游文档） |

| **用户要求写代码** | 规划助手应提示：编码须切换至编码任务，且以本文已批准条款为依据；**不在规划助手职责内代写** |
| **对话开场** | 规划类对话开始前 **必须先阅读本文**（§3.4「对话前必读」）；产出以本文增量为准，并在回复中说明写入的章节与版本 |

## 四、范围边界（明确不做的事）

以下事项 **不在毕设范围内**，出现相关需求一律拒绝或标记 P2，避免 scope creep：

| 禁止项 | 原因 |
|--------|------|
| 在 `分工/` 另建规划/修订类 Markdown | 违反 §3.3；一律并入本文专章 |
| 独立「用户管理」「系统设置」模块 | 与功能模块图 1—5—4 无关 |
| 调度员/审核员小程序版 | 开题方案与原型 B1 约束 |
| 长期前端 Mock、不联调后端 | 违背「先后端 → Web → 小程序」铁律 |
| 单页合并多个活动图节点 | 违背「一活动一界面」；**例外见 §十七**（受理材料审核合并核验与受理决定，经政务规范调研批准） |
| 跳过否支/↺ 回退路径 | 五条活动图均含判断与循环 |
| 微信真机 OAuth、WebSocket 实时推送 | 计划内 P2，答辩用账号密码即可 |
| 对接真实政务外网/生产 MySQL | 本地 H2 + seed 演示数据足够 |

### 17.3 判断分支与状态机

```
⟨材料是否齐全合规⟩
  否 → 需求退回通知 → 需求材料补充 ↺ 中试需求提交
  是 → ⟨是否同意受理⟩（同页完成，审核员一次操作）
         否 → 项目 CLOSED（不予受理）
         是 → 受理结果通知 → 受理回执签收 → 受理信息归档 → 查看受理进度 → 进入评估
```

| 状态 | 含义 | 修订说明 |
|------|------|----------|
| SUBMITTED | 已提交待受理 | — |
| ACCEPTING | 已登记，待材料审核 | 登记后由审核员一步审核 |
| RETURNED | 已退回待补充 | ↺ |
| ACCEPTED | 同意受理 | 由「材料齐全 + 同意」直接进入，**不再经 VERIFYING** |
| RECEIPTED / ARCHIVED | 已签收 / 需求段归档 | ARCHIVED 后 stage→EVALUATION |
| VERIFYING | （遗留枚举） | **新流程不产生**；兼容旧数据 |

#### 17.14.3 阶段下拉筛选（权威）

控件为 **下拉选择**（Web：`el-select`，占位符如「全部阶段」；小程序：阶段选择器），**默认「全部」**：

| 下拉选项文案 | 传参 `stageFilter` | 筛选条件（`trial_project`） |
|--------------|-------------------|------------------------------|
| **全部** | `ALL`（默认） | 企业名下全部可查看项目（同 §17.13，排除需求草稿） |
| **需求办理中** | `DEMAND` | `stage = DEMAND` |
| **评估中** | `EVALUATION` | `stage = EVALUATION` |
| **调度执行中** | `DISPATCH` | `stage = DISPATCH` |
| **反馈复核中** | `FEEDBACK` | `stage = FEEDBACK` |
| **档案统计中** | `ARCHIVE` | `stage = ARCHIVE` |
| **已结案** | `CLOSED` | `stage = CLOSED` |

#### 17.14.4 搜索（权威）

| 项 | 规范 |
|----|------|
| **控件** | Web：`el-input` + 搜索按钮（或回车触发）；小程序：搜索框 + 确认搜索 |
| **参数名** | `keyword`（可选，字符串；空或不传表示不按关键字过滤） |
| **匹配字段** | `trial_project.project_no`（项目编号）、`trial_project.title`（项目名称） |
| **匹配方式** | **模糊匹配**（包含即可，大小写不敏感）；编号、名称 **满足其一** 即命中（OR） |
| **与阶段筛选** | `stageFilter` 与 `keyword` **同时生效**（AND）：先限企业本人 + 阶段，再按关键字过滤 |
| **触发** | 点击「搜索」或回车；**清空搜索框并搜索** 视为 `keyword` 为空，恢复仅阶段筛选 |
| **分页** | 变更阶段或执行搜索时 **重置为第 1 页** |
| **空状态** | 有 `keyword` 且无结果：「未找到匹配的项目」；仅阶段筛选无结果：「该阶段暂无项目」 |
