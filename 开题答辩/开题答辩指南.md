## 〇、汇报逻辑主线（答辩叙述顺序）

> **原则：** 先讲 **业务**，再讲 **流程图**，再讲 **功能分解**，最后讲 **实现**。让评委先理解「中试是什么、中心在办什么事」，再看到图，再看到系统怎么做。

```
第 1 步  什么是中试？为什么要做中试？（**须引用 2024—2026 政策与统计数据**，见 §三）
         ↓
第 2 步  由此引出：中心的中试「服务」面临什么问题？本系统解决什么？
         ↓         （系统与中试的关系：管服务流程，不管试验操作本身）
第 3 步  核心业务活动是什么？→ 用 **5 张业务活动图** 讲清五条主线
         ↓
第 4 步  从各活动图中的 **核心活动节点** 归纳 → 推出 **功能模块图**
         ↓
第 5 步  展示功能模块图（1—5—4 结构）
         ↓
第 6 步  后续系统实现：架构、技术栈、开发计划
```

**答辩口径（建议统一说法）：**

- 「我先从 **中试业务** 讲起，再展示 **业务活动图** 梳理出的五条核心流程。」
- 「每条流程里的 **核心活动**（如需求提交、任务派发、报告复核）是系统必须支撑的关键步骤；把这些核心活动 **归纳、分解**，就得到了 **功能模块图** 里的二级、三级功能。」
- 「因此：**活动图回答『谁、按什么顺序、办什么事』；模块图回答『系统要提供哪些功能』。** 二者一一对应、互相校验。」

> **说明：** `开题方案.md` 正文按文档编写习惯写「模块图定稿后反推活动图」；**答辩口述** 建议采用本节顺序（先业务活动、再归纳模块），更符合评委理解路径，二者结论一致，不矛盾。

### 〇.1 整场答辩流程总览（17 页 · 8—10 分钟）

> **用途：** 进场前 **3 分钟** 过一遍；**正式稿背诵 §5.1**；翻 PPT 查 **§5.2**；限时 6 分钟用 **§六**。

**全程回扣一句话：** 本系统面向广州生产力促进中心，把企业申请中试服务到报告归档这条 **办理流程** 搬到线上；管 **服务协同与数据留痕**，**不管** 实验室里怎么做试验。

**总逻辑线：**

```
概念与政策 → 甲方痛点 → 现状与切口 → 目标与问题
    → 五条业务主线 → 五张活动图（流程）
    → 功能模块图（系统） → 技术架构与进度 → 总结致谢 → 教师提问
```

### 1.3 开题答辩要证明什么

老师主要想确认：

1. **选题有意义**——真实甲方、真实痛点，不是空套壳；
2. **边界清楚**——管的是中试 **服务流程**，不是实验室试验本身；
3. **分析到位**——角色、模块、流程说得清，有图支撑；
4. **方案可行**——技术成熟、工作量可控、16 周计划合理；
5. **后续能落地**——业务活动图能指导后续页面与接口设计。

**不必**在开题现场演示代码或数据库。

## 三、中试背景调研（须大量调研 + 时效性数据）

> **答辩要求：** 介绍中试不能只讲课本定义，须基于 **大量、多来源调研**，并引用 **近 2—3 年** 的政策与统计数据，体现选题 **有现实依据、有时代背景**。数据须 **注明出处与发布时间**，答辩 PPT 脚注或口述中点明「据××年××文件/报道」。

### 3.1 调研做什么（开题前/答辩前应完成）

| 调研维度 | 查什么 | 建议来源类型 | 时效性要求 |
|----------|--------|--------------|------------|
| **概念与链条** | 小试—中试—量产；中试在成果转化中的位置 | 政策文件术语、科技管理教材 | 概念可经典；**政策表述用最新版** |
| **国家层面** | 中试平台布局、建设目标、服务规模 | 国务院、工信部、科技部官网 | **2024—2026** 优先 |
| **广东省 / 广州市** | 地方实施方案、平台数量、重点产业 | 粤府办文件、广州市发改委/工信局 | **2024—2026** |
| **行业痛点** | 成果转化「最后一公里」、中试供给不足 | 工信部座谈会、人民日报/工信媒体 | **2024—2026** |
| **甲方单位** | 生产力促进中心职能、中试项目量、近期动态 | 中心官网、市工信局平台介绍 | **2024—2026**，中心数据可略早但须说明 |
| **信息化缺口** | 电话/Excel 协同 vs 平台化服务管理 | 开题调研、中心公开介绍中的业务描述 | 与痛点调研结合 |

**「大量调研」在答辩中的体现：**

- PPT 专设 **1—2 页「政策与数据背景」**，至少引用 **3 条国家级 + 2 条广州市/中心级** 时效信息；
- 自述稿第一步 **嵌入 2—3 个数字**（平台数量、项目数量、广州排名等）；
- 开题报告「国内外研究现状 / 选题背景」可展开写，答辩 **口述精华**；
- 准备 **调研参考文献表**（见 §3.3），老师问「数据哪来的」能答出具体文件。

### 5.1 完整演讲稿（正式稿 · 建议背诵本篇）

#### 一、开场与汇报结构

各位老师好！我是何至恒，学号 1230120215。我的毕业设计题目是《广州生产力促进中心中试服务管理系统的设计与实现》，指导教师是吕志峰老师。下面向各位老师汇报开题方案。

本次汇报分六个部分：中试背景与政策环境，中心业务现状与系统定位，文献调研与课题切入点，五条业务活动图，功能模块划分，以及技术路线、进度与成果。整体叙述顺序是 **先讲业务和流程，再讲系统功能，最后讲实现方案**——让各位老师先理解中心在办什么事，再看到系统怎么做。

---

#### 二、中试概念与政策背景

**什么是中试。** 科技成果产业化一般经历小试、中试、量产三个阶段。小试在实验室验证技术原理；**中试** 是在接近生产的条件下进行放大验证，形成中试报告，是连接实验室研发与工业化量产的关键环节。量产阶段设备与资金投入大、试错成本高，若跳过或弱化中试，技术风险将传导至量产环节。需要说明的是：本课题建设的系统，针对的是中试 **服务办理** 这一业务环节，帮助中心把企业申请、审核、派单到归档的流程搬到线上，**并不替代** 实验室内部的试验操作。

**政策与产业背景。** 近年来，国家与地方密集布局中试平台建设。2024 年，工业和信息化部印发《制造业中试创新发展实施意见》；广东省出台粤府办〔2024〕7 号，推动现代化中试平台体系建设。2025 年 5 月，广州市 12 家中试平台入选国家首批重点培育名单；据 2025 年公开报道，全国制造业中试平台已超过 2400 家，广州市 31 家平台上榜广东省首批名单、数量居全省第一。按照规划，到 2027 年将逐步形成多层次制造业中试服务网络。

政策推动的是 **能力建设**，平台与项目规模在快速扩大，中心对外 **服务办理量** 也将持续上升。与此同时，服务受理、派单、复核等流程若仍依赖传统协同方式，将难以匹配产业发展节奏——**流程管理信息化** 因此成为现实需求。以上数据出处见 PPT 页脚。

---

#### 三、中心现状、痛点与系统定位

落到具体甲方——**广州生产力促进中心**。中心提供研发设计、中试服务、技术解决方案等，增城中试基地在运行；公开信息显示，累计开展小试、中试服务项目 20 个以上，并于 2025 年入选高新技术成果产业化试点单位。业务参与方主要有四类：**企业项目负责人、中试技术人员、中试调度员、中试审核员**。

通过调研，归纳出四类痛点：**第一**，服务协同仍较多依赖电话、微信和 Excel，缺少统一入口和流程状态管理；**第二**，企业无法实时掌握办理节点，进度不透明；**第三**，中试报告复核环节容易遗漏，闭环不完整；**第四**，历史项目数据分散，难以开展台账维护与统计分析。

针对上述问题，本课题拟建设 **中试服务全流程管理系统**。请老师特别注意——**系统范围是中试服务办理流程，从申请到归档；不涵盖实验室试验操作。** 这是本课题最重要的边界界定。

---

#### 四、研究现状、切入点、目标与关键问题

**研究现状与切入点。** 国外实验室信息管理系统（LIMS）及项目管理系统较为成熟，但侧重实验数据与组织内部项目管理，对政府科技服务机构 **对外提供中试服务**、多端协同的场景支持不足。国内成果转化平台、中试基地门户较多，偏重信息发布和仪器预约，**审核—派单—复核—归档** 这条完整办理链往往缺失。

本课题的切入点在于：**其一**，有明确的甲方单位——广州生产力促进中心；**其二**，以需求、评估、调度、反馈、档案五条业务主线贯通服务全流程；**其三**，技术上采用 Web 分角色门户与微信小程序移动补充相结合的双端架构。课题边界明确为流程与数据信息化，不涉及试验执行；预期成果为可演示系统与毕业论文。

**研究目标与关键问题。** 总目标是设计并实现一套中试服务全流程信息化管理系统，覆盖上述五条业务链路。具体而言：**目标一**，实现五条业务链线上贯通，节点状态可查询、全过程可追溯；**目标二**，建设 Web 分角色门户与微信小程序移动补充，后端提供统一的 RESTful 接口，保障双端数据一致。

需要重点攻克的关键问题有两类：**一是** 多角色权限划分与责任边界问题，拟采用基于角色的访问控制（RBAC）与工作流引擎加以解决；**二是** 多端进度同步、报告复核业务闭环，以及历史数据入库与统计分析问题。上述问题将指导后续的详细设计与编码实现。

---

#### 五、五条核心业务与业务活动图

**业务主线概览。** 中心对外中试服务按五条主线组织，顺序为：**需求管理 → 评估管理 → 调度管理 → 反馈管理 → 档案管理**。其中，需求与评估是服务入口，调度与反馈是过程执行与质量把关，档案是统计分析与结案收尾。五张业务活动图串起来，构成从企业提交需求到项目结案的完整服务链，也与功能模块图的五个二级模块 **一一对应**。

下面结合五张业务活动图，说明各环节的流程要点。

**图 3-1 · 中试需求管理（服务链第一环，10 个节点）。** 读图时请注意：泳道上方是岗位、下方是具体活动；菱形表示判断分支；↺ 表示退回补充或重新办理。本图三泳道分别为企业、调度员和审核员。主路径为：企业确认并提交需求，调度员受理登记，审核员核验材料；材料合规且同意受理后，通知企业，企业签收，调度员完成受理信息归档，企业可随时查看受理进度。若材料不齐，退回补充后重新提交；若不同意受理，流程终止。该图核心活动对应中试需求管理模块的四项三级功能：需求提交、材料补充、受理登记、退回通知。

**图 3-2 · 中试评估管理（12 个节点）。** 本环节承担条件判定与可行性审查，决定项目 **能否进入调度环节**。主路径为：调度员开展前置核查、条件评估与资源核定，审核员进行技术可行性审查并出具评估结论；企业签收结论、填写评估意见反馈后，调度员归档，企业可查看评估结论。不具备中试条件或技术不可行时，分别通过材料补充循环退回。对应评估管理模块四项功能：条件评估、可行性审查、资源核定、评估结论出具。

**图 3-3 · 中试调度管理（11 个节点，业务中枢）。** 本环节负责资源匹配、任务派发与执行跟踪。三泳道为调度员、技术人员和企业。主路径为：调度员完成资源匹配与任务派发，技术人员接收确认并填报执行进度，进度通报至结果确认与归档。资源匹配失败时循环重匹配；执行出现异常时，启动异常任务重派机制。对应调度管理四项功能：资源匹配、任务派发、进度跟踪、异常重派。

**图 3-4 · 中试反馈管理（11 个节点，质量关口）。** 本环节负责试验结果提交、报告审核与复核确认。技术人员提交试验结果，审核员完成报告审核与复核确认，企业反馈意见。审核或复核不通过时，退回修改试验结果后重新提交，图中设有三处判断节点，体现完整的质量把关与回退机制。对应反馈管理四项功能：结果提交、报告审核、复核确认、报告归档。

**图 3-5 · 中试档案管理（9 个节点，服务链收尾）。** 本环节负责台账维护、统计分析与简报生成。主路径为：台账维护、资料归集、周期统计、成功率分析、简报生成与审核，最终完成档案归档。台账不完整或统计数据不可用时，循环补录后重新统计。

五张图完整串联，即：**需求受理 → 条件评估 → 任务调度 → 结果反馈与复核 → 档案统计结案**。

#### 六、功能模块设计

业务活动图回答了 **"谁、按什么顺序、办什么事"**；功能模块图则回答 **"系统要提供哪些功能"**。二者一一对应、互相校验。

从各活动图中的核心活动节点归纳分解，得到系统功能模块的 **1—5—4** 结构：一个系统、五个二级模块、每个模块四个三级功能，共计 **二十项**，与开题方案 6.3 节模块清单一致。五个二级模块——中试需求管理、中试评估管理、中试调度管理、中试反馈管理、中试档案管理——分别与五张业务活动图对应。模块划分遵循高内聚、低耦合原则，便于后续按模块分阶段开发与测试。**功能清单不是凭空罗列菜单，而是从业务流程推导出来的。**

#### 七、技术方案、当前进度与实施计划

**技术架构。** 系统采用分层架构：表现层为 Web 端与微信小程序；接口层采用 RESTful API，前后端分离；业务层基于 Spring Boot 与 MyBatis；数据层采用 MySQL。Web 端部署统一的 Vue 前端，用户登录后按 RBAC 机制路由至企业门户、中心管理端或技术人员端；微信小程序面向外出与现场场景，支持进度查询、消息通知与精简填报，复杂审核与派单操作仍在 Web 端完成。所选技术栈成熟稳定，本科毕业设计工作量可控。

**当前进度与计划。** 截至目前，五张业务活动图与功能模块图均已完成建模出图，需求分析阶段的核心成果已经就绪。可运行系统将于开题通过后进入设计与开发阶段，计划在 **十六周** 内完成 Web 端、小程序端系统及毕业论文撰写，优先打通 **从需求受理到任务调度** 的可演示业务闭环，再逐步扩展至五条主线完整闭环。

## 八、常见问题与参考回答

### 8.1 选题与业务

**Q：中试是什么？为什么要做中试？**  
A：中试是 **中间试验**，处于小试与量产之间，在接近生产的条件下做放大验证，检验工艺、质量、成本和设备适配性，并形成中试报告支撑量产决策。跳过或做不好中试，技术风险会传导到量产。本系统不替代中试试验本身，而是管理 **中心对外提供中试服务** 的办理流程。

**Q：系统和中试这个词怎么关联？**  
A：中心的核心业务之一是 **中试服务**——帮企业完成从中试申请到报告归档的全过程。系统管的是这套 **服务管理流程**（申请、审核、派单、执行跟踪、复核、归档），所以叫「中试服务管理系统」；不是 LIMS，不管反应釜怎么控温。

**Q：中试和本系统的关系是什么？**  
A：中试是科技成果放大验证的试验阶段；本系统不替代试验操作，而是管理广州生产力促进中心对外提供中试服务的 **申请、审核、派单、跟踪、复核、归档** 全流程。

**Q：为什么选广州生产力促进中心？数据从哪来？**  
A：**背景调研数据** 来自工信部、广东省/广州市公开政策与报道（见 §3.2、§3.3），答辩时注明文件年份。**系统开发测试** 使用模拟业务数据，不依赖政府内部未公开数据。中心职能与「中试项目 20+」等来自市工信局平台公开介绍。

**Q：你的调研充分吗？数据是否过时？**  
A：开题阶段对 **2024—2026 年** 国家及广州市中试政策、平台数量、中心动态做了专项检索；PPT 中关键数字 **标注发布年份**。正式论文撰写前会 **再次核对** 最新文件；若统计口径更新，以最新官方发布为准。

**Q：中试平台这么多，为什么还要做你的系统？**  
A：国家与广州强调的是 **中试验证能力建设** 和 **平台布局**；中心还要解决 **对外服务流程** 的受理、派单、复核与归档。平台数量增长后， **服务办理量** 上升，传统协同方式更容易成为瓶颈——本系统针对的是 **服务管理信息化**，与平台建设互补而非重复。

**Q：和现有科技成果转化平台有什么区别？**  
A：现有平台多侧重成果展示和需求发布；本系统聚焦 **服务办理闭环** 和 **双端协同**，强调审核—评估—派单—执行—复核—归档。

### 8.3 技术与工作量

**Q：Web 和小程序怎么协同？**  
A：Web 是 **办公主渠道**——同一 Vue 前端按 **登录角色** 展示企业门户、技术人员门户或中心管理端；小程序是企业、技术人员的 **移动补充**（外出查进度、现场填报）。调度员、审核员仅在 Web 中心管理端操作。两端 **共用 Spring Boot 后端和 MySQL**，RESTful API 同步状态。

**Q：本科能不能做完？**  
A：技术栈成熟，无复杂算法；按模块分阶段实现，优先完成五条主流程的可演示闭环，统计简报等辅助功能可后置，工作量可控。

**Q：有什么创新点？**  
A：（择一两条说，不宜夸大）① 面向 **生产力促进中心** 这一具体公益科技服务机构的中试 **服务管理** 闭环；② **Web 分角色门户 + 小程序移动补充**，与业务活动图岗位一一映射；③ 流程含完整 **回退与归档** 机制，非简单 CRUD。

### 8.4 术语速查（答辩可能被问）

#### RBAC（Role-Based Access Control · 基于角色的访问控制）

**缩写：** **R**ole-**B**ased **A**ccess **C**ontrol  

**含义：** 不逐个用户单独配权限，而是先定义 **角色**，再给角色分配 **能做什么**；用户登录后按 **所属角色** 决定可见菜单、可操作按钮、可调用的接口。

```
用户 → 绑定角色 → 角色拥有权限 → 允许 / 拒绝操作
```

**在本课题中的用法：**

| 角色 | 大致权限 |
|------|----------|
| 企业项目负责人 | 提需求、补材料、查进度、反馈意见 |
| 中试调度员 | 受理登记、条件评估、资源匹配、派单、进度督办 |
| 中试审核员 | 审材料、审可行性/报告、复核确认、归档把关 |
| 中试技术人员 | 接任务、填报进度、提交试验结果 |

Web 端 **同一套 Vue 前端**，登录后按 RBAC **路由到不同门户**（企业 / 中心管理 / 技术人员）；调度员、审核员仅在中心管理端操作，小程序不做复杂审核派单。

**与「普通登录」的区别：** 仅有登录只证明「你是谁」；RBAC 进一步规定 **不同角色能干什么**，避免企业账号去派单、技术人员去终审报告。

**与工作流的关系：** **RBAC** 管 **权限边界**（谁能点哪个功能）；**工作流引擎** 管 **业务顺序**（需求先到调度员受理、再到审核员核验、再进入派单）。答辩第 7 页「关键问题一」即二者配合解决多角色协同。

**答辩一句话：**  
RBAC 就是按调度员、审核员等 **角色** 分配系统权限，保证 **各干各的、责任清楚**。
