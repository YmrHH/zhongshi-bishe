## 三、核心设计原则

### 3.1 一活动一主界面（硬性）

每个业务活动圆角矩形对应 **至少一个主页面**（或同一页内 **独立 Tab / 步骤条**），节点名称采用 **动宾结构**，宜直接作页面标题或主按钮文案。

| 原则 | 说明 | 反例 |
|------|------|------|
| 不合并 | 「中试需求提交」与「需求材料补充」不得挤在同一复杂表单页 | 单页承担整条受理链 |
| 不遗漏 | 补充节点（确认、签收、通知、归档）也须有对应界面或明确 UI 步骤 | 仅有核心 4 节点无签收页 |
| 可回退 | 循环回退（↺）须能在原型中 **再次进入** 目标活动页 | 修改后无法回到提交页 |

### 3.2 按泳道汇总导航（硬性）

同一泳道内活动 **自上而下** = 该岗位在该模块下的 **菜单 / 待办操作顺序**。

```
泳道标题（岗位）
    ↓
活动 1 → 页面 A
活动 2 → 页面 B
    …
跨泳道水平连线 → 上一页「提交/通知」→ 下一岗位「待办列表」出现新项
```

### 3.3 判断分支 → 分支界面（硬性）

| 判断结果 | 原型表现 |
|----------|----------|
| 「是」支 | 进入下一活动对应页面；列表状态变为「已通过 / 待下一环节」 |
| 「否」支 | 退回页、补充材料页、修改页或终止提示；须标注 **回退目标页** |
| 循环 ↺ | 原型连线标注「修改后重新提交」，目标页与活动图一致 |

### 3.4 双端分工（硬性）

| 端 | 职责 | 本选题角色 |
|----|------|------------|
| **Web** | 办公主渠道；中心管理、企业门户、技术人员门户 | 四类角色均可登录（按账号路由） |
| **微信小程序** | 移动补充；外出、现场等不宜用电脑场景 | **仅** 企业项目负责人、中试技术人员 |

**端别标注规则：**

- 中心岗位泳道括注 **（Web）**——原型 **仅出 Web 版**，不出小程序版。
- 企业、技术人员泳道括注 **（Web/小程序）**——**Web 为主稿必出**；小程序出 **对应核心页 + 查看类页**，不重复中心管理功能。

**协同要求（与《选题规范.txt》§五一致）：**

- 小程序提交 → Web 端待办出现（如企业提交需求 → 调度员「待受理」）。
- 两端 **共用后端 API**，原型须标注 **状态同步**（如「提交成功后 Web 待办 +1」）。

### 3.5 分角色门户（硬性）

Web 端为 **同一 Vue 工程、按登录角色路由**，原型须分 **三套 Web 门户** 出图：

| 门户 | 角色 | 风格定位 |
|------|------|----------|
| 中心管理端 | 中试调度员、中试审核员 | 待办驱动 + 审核表单 + 统计看板 |
| 企业门户 | 企业项目负责人 | 申请填报 + 进度跟踪 + 签收反馈 |
| 技术人员门户 | 中试技术人员 | 任务接收 + 进度填报 + 结果提交 |

调度员与审核员可 **共用中心管理端壳体**（顶栏/侧栏一致），通过 **菜单权限** 区分功能，原型中可用 **角色切换说明** 标注。

## 七、页面设计规范

### 7.1 通用布局（Web）

| 区域 | 要求 |
|------|------|
| 顶栏 | 系统名称（含「广州生产力促进中心中试服务管理系统」简称）、当前角色、消息、用户 |
| 侧栏 | 按 §5.4 五模块导航；当前模块高亮 |
| 主内容区 | 面包屑 + 页面标题（与活动节点名一致或略作 UI 化） |
| 底栏（可选） | 版权、版本号 |

**推荐组件库（与实现一致）：** Element Plus 风格 — 表格、表单、步骤条、对话框、标签（状态色）。

### 7.2 页面类型与必含元素

| 类型 | 典型页面 | 必含元素 |
|------|----------|----------|
| 列表 + 待办 | 需求受理工作台、报告审核页 | 筛选（状态/时间）、表格、操作列（办理/退回）、分页 |
| 表单填报 | 需求填报页、结果提交页 | 分组表单、附件上传、校验提示、提交/暂存 |
| 审核 / 确认 | 材料核验页、复核确认页 | 业务详情只读区、审核意见、通过/退回按钮 |
| 签收 / 通知 | 受理回执签收页、派单通知页 | 通知内容、确认签收按钮、关联业务单号 |
| 进度 / 详情 | 受理进度详情页、进度查看页 | **步骤条**（与活动图主路径一致）、当前状态、历史记录 |
| 统计 / 分析 | 周期统计页、成功率分析页 | 筛选条件、图表区、导出按钮 |
| 归档 | 各模块 XX 归档页 | 归档清单、确认归档、归档后只读 |

### 7.3 状态标签（全系统统一）

| 状态码 | 展示文案 | 色建议 | 适用场景 |
|--------|----------|--------|----------|
| draft | 草稿 | 灰 | 未提交需求 |
| pending | 待受理 / 待审核 / 待接收 | 橙 | 待办列表 |
| processing | 办理中 / 执行中 | 蓝 | 流程中段 |
| returned | 已退回 / 待补充 | 红 | 否支、循环回退 |
| completed | 已完成 / 已归档 | 绿 | 成功结束 |
| rejected | 已终止 | 灰红 | 不同意受理等汇 ◉ 支路 |

### 7.4 小程序设计约束

| 项目 | 要求 |
|------|------|
| 页面深度 | 单流程 **≤ 4 层**；过深流程在 Web 完成 |
| 必做页面 | 需求填报、材料补充、进度查看、任务接收、进度填报、结果提交 |
| 不做 | 中心审核、派单、统计、档案维护（仅 Web） |
| 交互 | 下拉刷新列表；提交成功后 `wx.showToast` + 跳转详情 |

### 7.5 视觉与品牌

| 项目 | 建议 |
|------|------|
| 主色 | 科技蓝 `#1890FF` 或 Element 默认蓝 |
| 辅色 | 成功 `#67C23A`、警告 `#E6A23C`、危险 `#F56C6C` |
| 字体 | 中文：微软雅黑 / PingFang SC；英文/数字：Arial |
| 一致性 | 同一门户内按钮位置、表格样式、对话框交互 **保持一致** |
