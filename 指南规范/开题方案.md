# 毕业设计开题方案

**题目：** 广州生产力促进中心中试服务管理系统的设计与实现  
**专业：** 软件工程  
**系统架构：** Web 端（按角色分页面）+ 微信小程序（移动补充）  
**编制日期：** 2026年6月

---

## 一、选题背景与意义

### 1.1 选题背景

科技成果转化是广州建设国际科技创新中心的重要支撑。科技成果从实验室研发走向规模化生产，通常需经历 **小试、中试、量产** 三个阶段，其中 **中试（中间试验）** 是验证技术能否放大试产、能否稳定生产的关键环节。

**广州生产力促进中心** 是广州市政府批准组建的公益性、综合性科技服务机构，技术服务板块明确包含 **研发设计、中试服务、技术解决方案、科技成果交易** 等内容。中心依托增城科研中试生产基地，推动组建中试服务联盟，累计开展小试、中试服务项目超 20 个，在广州市中试服务领域具有明确职能定位。

当前，中心的中试服务涉及 **企业项目负责人、中试技术人员、中试调度员、中试审核员** 等多方协同，业务环节包括需求提交、受理审核、条件评估、资源匹配、任务派发、试验执行、结果反馈、报告复核与归档等。现有工作模式较多依赖 **电话沟通、微信群协调、纸质材料和 Excel 台账**，存在以下问题：

1. 中试需求状态不透明，企业难以实时掌握办理进度；
2. 审核、派单、执行、复核责任链条不清晰；
3. 任务超时缺乏系统督办，反馈容易滞后；
4. 中试报告复核闭环不完整，历史项目难以沉淀；
5. 中试项目数据分散，无法支撑服务效率分析与决策优化。

因此，建设一套面向广州生产力促进中心的 **中试服务管理系统**，实现 **Web 分角色门户与微信小程序移动补充** 协同，具有明确的现实需求和应用价值。

### 1.2 研究意义

**理论意义：** 将软件工程中的需求分析、业务建模、系统设计、数据库设计与双端协同开发方法，应用于科技服务机构的中试服务管理场景，探索 **"政府科技服务机构 + 双端业务闭环"** 的信息化实现路径。

**实践意义：**

1. 为广州生产力促进中心提供中试服务全流程数字化管理工具；
2. 提升中试需求受理、资源调度、任务执行、报告复核的协同效率；
3. 实现中试项目过程可追溯、结果可复核、数据可沉淀；
4. 为本科毕业设计提供完整、可落地的软件工程实践案例。

## 二、国内外研究现状

### 2.1 国外研究现状

欧美等发达国家较早建立科技成果转化服务体系，部分高校、国家实验室和生产力促进机构通过 **LIMS（实验室信息管理系统）**、**Project Management System** 等工具管理研发与中试项目。这类系统侧重实验数据管理、项目进度跟踪与文档归档，但多面向单一实验室或企业内部，较少结合 **政府科技服务机构对外提供中试服务** 的双端协同场景。

### 2.2 国内研究现状

国内近年来围绕 **科技成果转化、中试平台建设、技术转移服务** 出台多项政策。部分地区建设科技成果转化公共服务平台、中试基地管理平台，功能多集中在 **成果展示、需求发布、仪器预约、政策查询** 等方面。

现有研究与系统存在以下不足：

1. 多数平台偏重信息发布或资源展示，缺少 **审核—评估—派单—执行—复核—归档** 完整业务闭环；
2. 较少针对 **生产力促进中心** 这一具体甲方设计系统，通用平台难以体现机构职责特征；
3. 管理端与移动端协同不足，企业申请端与内部管理端往往割裂；
4. 面向本科毕设可实现的、业务流程清晰的中试 **服务管理** 系统案例相对较少。

### 2.3 本课题切入点

本课题以 **广州生产力促进中心** 为具体甲方，聚焦其 **中试服务管理** 核心业务，设计 **Web 分角色门户 + 微信小程序移动补充** 协同系统，补齐现有研究中"业务流程闭环不完整、多端协同不足、甲方场景不具体"的不足。

## 三、研究目标与内容

### 3.1 研究目标

设计并实现 **广州生产力促进中心中试服务管理系统**，完成中试服务从 **需求提交 → 受理审核 → 条件评估 → 资源调度 → 试验执行 → 结果反馈 → 报告复核 → 项目归档** 的全流程信息化管理，满足以下目标：

1. 企业、技术人员可在 **Web 端**（按账号进入不同门户）完成需求提交、任务处理、进度查看与结果反馈；**微信小程序** 作为外出、现场等移动场景补充；
2. 中心工作人员可在 Web 管理端完成审核、评估、派单、督办、复核与统计；
3. 系统实现 4 类角色、5 项核心业务活动、5 个功能模块的业务闭环；
4. 系统适合本科阶段独立完成开发与论文撰写。

### 3.2 研究内容

1. **需求调研与分析**  
   调研广州生产力促进中心中试服务业务流程，明确甲方职责、服务对象、业务痛点与系统边界。

2. **业务建模**  
   在已定稿功能模块图基础上，反推 **5 个** 核心业务活动（与 5 个二级模块对应），绘制业务活动图（每图 ≥3 角色、**≥8 活动节点**、复杂模块 **9—12 个**；含判断与循环；覆盖全部三级功能；**作为双端页面设计依据**）。

3. **系统设计**  
   完成系统总体架构、功能模块、数据库、接口与双端页面设计。

4. **系统实现**  
   实现 Web 分角色门户、微信小程序移动补充及后端服务，完成核心业务功能。

5. **系统测试**  
   开展功能测试与业务流程测试，验证系统是否满足开题目标。

6. **论文撰写**  
   按照学校毕业设计规范完成论文各章节撰写。

## 四、拟解决的关键问题

1. 如何将广州生产力促进中心中试服务真实业务流程抽象为可实现的系统功能？
2. 如何实现 **Web 分角色门户** 与 **微信小程序** 之间的真实业务流转与状态同步？
3. 如何设计中试需求审核、评估派单、报告复核等环节的闭环与异常回退机制？
4. 如何设计数据库模型以支撑中试需求、评估、任务、报告、档案等核心业务数据？
5. 如何在本科工作量范围内保证系统完整性与可演示性？

## 五、业务分析

### 5.1 甲方与系统定位

| 项目 | 说明 |
|------|------|
| 精确甲方 | 广州生产力促进中心 |
| 系统定位 | 中试服务全流程管理平台 |
| 系统不是什么 | 不是中试试验本身，不是纯展示网站，不是普通审批后台 |

### 5.2 中试概念简述

**中试（中间试验）** 是科技成果从实验室研发走向规模化生产之间的关键验证阶段，主要验证工艺放大可行性、产品质量稳定性、成本可控性、设备场地适配性及行业标准符合性。中试完成后形成 **中试报告**，为企业是否投入量产提供依据。

本系统管理的是 **中试服务流程**，而非替代中试试验本身。

### 5.3 服务对象

| 序号 | 角色 | 使用端 | 主要职责 |
|------|------|--------|----------|
| 1 | 企业项目负责人 | Web 端 + 微信小程序 | **办公以 Web 企业门户为主**（需求提交、材料补充、进度查看、结果反馈）；外出可用小程序 |
| 2 | 中试技术人员 | Web 端 + 微信小程序 | **办公以 Web 技术人员门户为主**（接收任务、进度填报、结果提交）；现场/外出可用小程序 |
| 3 | 中试调度员 | Web 管理端 | 受理审核、条件评估、资源匹配、任务派发、进度督办 |
| 4 | 中试审核员 | Web 管理端 | 审核材料、复核结果、确认报告、项目归档 |

> **端别说明：** 同一套 Vue Web 前端按 **登录账号角色** 路由至不同页面（企业门户 / 技术人员门户 / 中心管理端）；小程序 **不替代** Web 办公入口，仅作移动补充。

### 5.4 业务痛点

| 痛点 | 现状 | 系统改善 |
|------|------|----------|
| 协同方式落后 | 电话、微信群、Excel | Web + 小程序在线协同 |
| 状态不透明 | 企业不知进度 | 全程状态可查 |
| 责任不清晰 | 派单、执行、复核脱节 | 角色职责明确 |
| 闭环不完整 | 报告复核易缺失 | 审核—执行—复核—归档闭环 |
| 数据无沉淀 | 历史项目难统计 | 台账与统计分析 |

### 5.5 设计顺序说明（功能模块图反推业务活动图）

本选题 **功能模块图（§6.3 图 2-1）已定稿**，共 5 个二级模块、20 个三级功能。业务活动图按《UML图规范.md》要求 **由功能模块图反推**：**一模块一图**，共 **5 张**核心业务活动图；泳道为 **具体岗位名称**；每张图 **≥8 个活动节点**（**4 个核心节点** + **补充节点**），协作链长、双端交互多的模块 **可增至 9—12 个**（参照 `模板图/业务活动图.png` 约 12—16 个活动的密度），以便 **后续按活动节点设计 Web 页面（主）与小程序页面（辅）**。

#### 泳道、角色与业务活动的关系（与模板一致）

垂直泳道图采用 **「上为角色、下为活动」** 的读法，与模板图中「患者 / 医生 / 随访人员」等画法 **一致且合理**：

| 图元位置 | 含义 | 本选题示例 |
|----------|------|------------|
| 泳道 **标题行** | **谁** 参与本流程 | 企业项目负责人、中试调度员（Web） |
| 标题 **下方** 圆角矩形 | **该角色执行的操作步骤** | 「中试需求提交」「需求受理登记」 |
| **同列** 自上而下 | 该角色在流程中的 **时间顺序** | 先确认 → 再提交 → 再补充材料 |
| **跨列** 水平连线 | **岗位间业务交接** | 企业提交 → 调度员受理；技术人员填报 → 企业查看 |

**归属原则：** 活动框画在 **主要执行者** 所在泳道；「查看/签收」画在 **实际打开界面操作** 的角色泳道（企业在 Web 门户或小程序查看进度 → 画在企业列，而非中心管理列）。**禁止** 因某功能属于 Web 端而将活动画在「Web」抽象泳道——须写 **具体岗位名**。

#### 节点规模与页面设计导向

```
三级功能（核心节点，每图 4 个）
        ↓ 拆分为可独立成页的步骤
补充节点（确认 / 校验 / 通知 / 签收 / 预览 …）
        ↓ 按泳道汇总
岗位 → 活动序列 → Web 页 / 小程序页（§5.10）
```

- **下限：** 每图 **≥8** 个活动节点；**不足则难以支撑双端页面拆分**。
- **建议：** 需求受理、调度执行、反馈复核等 **复杂度高** 的模块 **9—12** 个；台账统计等 **8—9** 个即可。
- **参照：** `模板图/业务活动图.png`（随访预警，约 **16** 活动，多段跨泳道）；`模板图/业务活动图1.jpg`～`业务活动图4.jpg`（食物管理 **4 例**，每图 **8—11** 活动，**申请—审核—执行—反馈—归档** 骨架清晰）。

**从模板提炼的节点粒度原则（本选题须遵守）：**

1. **一动作一节点：** 能独立成页的步骤 **不合并**（如「出库」与「发货」、「签收」与「反馈」分节点；对应本系统「提交」与「校验」、「派发」与「接收确认」）。
2. **三类骨架选用：**
   - **A 申请型**（3-2/3-3）：发起方 ● 开始 → 管理方 **审核** ↺ → 执行方 **出库/执行链** → 发起方 **签收+反馈** → 管理方 **反馈审核** ↺ → **归档** → ◉  
   - **B 交接型**（3-1）：发起 **提交→发货** → 中间方 **接收→清点** ↺ → 专业方 **检验→分流** → **合并反馈** → **归档** → 发起方 **通知** → ◉  
   - **C 运维型**（3-4/随访）： **巡检/匹配** ↺ → **申请/派发** → **确认** → **盘查/判断** → 执行链 → **反馈审核** ↺ → **归档** → ◉  
3. **两次审核：** 材料/报告类流程宜有 **初审 + 结果/反馈复审**（本系统：报告审核 + 复核确认；可增 **反馈审核** 节点对应模板「反馈审核」页）。
4. **归档与通知：** 成功路径末段宜有 **归档**（管理岗泳道）+ 发起方 **查看/签收/通知**（用户泳道），再汇 ◉；禁止业务无归档直接结束。
5. **否支语义：** 模板「不通过」均 **回到可修改活动**（非直接 ◉）；仅 **业务终止**（如不具备条件、不同意受理）才 **汇 ◉**。
6. **节点命名：** 动宾结构、**6 字左右**，与三级功能或页面标题一致（如「需求受理登记」「试验结果提交」）。

#### 文字版绘制约定

| 图元 | 要求 |
|------|------|
| `┌ ┐ └ ┘ │ ─` | 外边框（左、右、下）+ 泳道分隔；内容不越界 |
| [活动] | **1 进 1 出**；多路仅在 **入线汇点** 合并，汇前 **分线独立** |
| ⟨判断⟩ | **1 进 2 出**；**否/是** 分别从 **不同顶角** 引出 |
| `─` `│` | 拐点 **直接转折**，**不写箭头**；**能直连的不拐弯** |
| `→` `↓` | **仅** 在连线 **进入节点** 处标 1 个，**指向明确** |
| 多线 | **禁止粘合重合**；每条线全程 **独立可辨** |
| ↺ | 接在目标活动 **入线** 上；走左/右 **独立通道** |
| ● / ◉ | 开始 **实心黑圆** / 结束 **靶心圆**；各 **1 个**；结束多路仅在 **◉ 入线汇点** 合并 |
| 绝对禁止 | **穿活动框**、**穿判断框**、**线条交叉**、**双向箭头**、**无必要绕圈** |

#### 连线独立与路由原则

1. **最短路径 / 直连优先：** 同泳道 → 竖直 **1 段**；同高跨泳道 → 水平 **1 段**；否则 **最多 2 段**（一横一竖或两竖一横）；**禁止** 为「好看」绕外框、绕远路。
2. **汇 ◉ 规则：** 终止支路 **能直插 ◉ 则直插**；多路汇入 ◉ 时，汇点 **紧贴 ◉ 上端**（短公共入线），**禁止** 沿左侧/右侧外框长途绕行后再汇 ◉。
3. **禁止双向箭头：** 仅 **单向** 流转；循环为 **单向回退**，不得出现 ↔。
4. **分线独立：** 每条线全程可辨；禁止长距离粘合。
5. **错层汇点：** 多路汇入活动/◉ 时，汇前 **分线独立**，汇点 **尽量靠近目标节点**。
6. **分侧回路：** 循环走左/右通道时，取 **不穿框、不交叉的最短侧向路径**。
7. **生图后必检：** 对照本图「连线路由表」+《UML图规范.md》§十九 逐条勾选。

### 5.10 活动节点与双端页面映射原则

业务活动图定稿后，按本表思路 **由活动节点反推页面**，避免 Web/小程序设计脱离流程（参照模板图「一列一角色、一步一活动」的粒度）。

#### 映射规则

| 规则 | 说明 | 示例 |
|------|------|------|
| 一活动一主界面 | 每个圆角矩形对应 **一个主页面**（或同一页内 **独立步骤/Tab**） | 「中试需求提交」→ Web 企业门户「需求填报」页（小程序可补充） |
| 按泳道汇总 | 同一泳道内活动 **自上而下** = 该岗位 **菜单/待办顺序** | 调度员：受理登记 → 退回通知 → … |
| 跨泳道连线 | 上一活动 **提交/通知** → 下一岗位 **待办列表** 出现新项 | 企业提交 → 调度员「待受理」 |
| 判断分支 | 「否」支常对应 **退回/修改/重填页**；「是」支进入 **下一活动页** | 材料不齐 → 补充材料页 |
| 循环回退 | 须能 **再次打开** 目标活动对应页面 | 修改后重新进入「试验结果提交」 |
| 端别 | 办公场景优先 **Web**（企业/技术人员/中心分角色页面）；**小程序** 作移动补充，与 §5.3 一致 | 企业提交 → Web 企业门户（主）+ 小程序（辅） |

## 六、系统设计

### 6.1 系统总体架构

系统采用 **B/S Web + 微信小程序** 架构：Web 端为 **办公主渠道**（按登录角色展示企业门户、技术人员门户或中心管理端），小程序为 **移动补充**。

```
┌─────────────────────────────────────────────────────┐
│                    表现层                            │
│  ┌────────────────────────────┐  ┌─────────────────┐ │
│  │  Web 端（Vue + Element）    │  │  微信小程序      │ │
│  │  按账号分角色页面：         │  │  企业/技术人员   │ │
│  │  · 中心管理（调度/审核）    │  │  移动办公补充    │ │
│  │  · 企业门户                │  │                 │ │
│  │  · 技术人员门户            │  │                 │ │
│  └─────────────┬──────────────┘  └────────┬────────┘ │
└────────────────┼──────────────────────────┼───────────┘
                 │      RESTful API         │
┌────────────────┼──────────────────────────┼───────────┐
│                ▼         业务层           ▼           │
│  ┌─────────────────────────────────────────────┐     │
│  │  Spring Boot 后端服务                        │     │
│  │  需求管理 / 评估管理 / 调度管理 / 反馈管理   │     │
│  └──────────────────────┬──────────────────────┘     │
└─────────────────────────┼────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────┐
│                    数据层                            │
│              MySQL 关系型数据库                      │
└─────────────────────────────────────────────────────┘
```

**技术选型（建议）：**

| 层次 | 技术 |
|------|------|
| Web 前端 | Vue 3 + Element Plus |
| 小程序端 | 微信原生小程序 / uni-app |
| 后端 | Spring Boot + MyBatis |
| 数据库 | MySQL 8.0 |
| 服务器 | Tomcat / 内嵌 Spring Boot |
| 开发工具 | IDEA、微信开发者工具 |

## 七、可行性分析

### 7.1 技术可行性

系统采用成熟的 Spring Boot + Vue + 微信小程序 + MySQL 技术栈，开发文档与社区资源丰富，适合本科毕业设计实现。系统不涉及复杂算法、大模型训练或专业硬件设备，技术风险可控。

### 7.2 经济可行性

开发可使用开源框架与免费开发工具，部署可使用本地服务器或云服务器演示，无需额外采购商业软件或硬件。

### 7.3 操作可行性

系统界面按角色划分功能，企业端以申请和反馈为主，管理端以审核、派单、复核为主，操作流程与现有业务习惯一致，易于理解和使用。

### 7.4 数据可行性

系统可使用 **模拟数据、用户填报数据和业务流程数据** 完成开发与测试，不依赖不可获取的政府内部核心数据，数据获取可行。


各位老师好，我是 ______，毕业设计题目是 **广州生产力促进中心中试服务管理系统的设计与实现**。

**第一，选题背景。** 科技成果转化需经历小试、中试、量产三个阶段。中试是验证技术能否放大试产的关键环节。广州生产力促进中心是市政府批准组建的科技服务机构，技术服务板块明确包含中试服务，累计开展小试、中试项目超 20 个。当前中心的中试服务涉及企业、技术人员、调度员、审核员多方协同，但较多依赖电话、微信群和 Excel，存在状态不透明、责任不清晰、复核闭环不完整等问题。

**第二，研究目标。** 本课题拟设计并实现一套 **Web 分角色门户 + 微信小程序移动补充** 的中试服务管理系统，覆盖 **需求提交、受理审核、条件评估、资源调度、试验执行、结果反馈、报告复核、项目归档** 全流程。系统管理的是中试 **服务流程**，不替代实验室内的试验操作本身。

**第三，业务分析。** 系统面向 4 类角色：企业、技术人员 **办公以 Web 门户为主**（不同账号登录对应不同页面），外出可用小程序；调度员、审核员在 Web 中心管理端审核、评估、派单、复核与归档。我按 5 个二级模块反推了 **5 张** 核心业务活动图：需求管理、评估管理、调度管理、反馈管理、档案管理；每张图不少于 **8 个活动节点**（含三级功能核心节点与必要补充节点），含判断与循环流程，20 个三级功能全部覆盖。

**第四，技术路线。** 后端采用 Spring Boot + MyBatis，Web 端 Vue 3 + Element Plus（按角色路由），小程序作移动补充，数据库 MySQL。计划在 16 周内完成需求分析、业务建模、系统开发、测试和论文撰写。

**第五，预期成果。** 一套可演示的 Web + 小程序系统、一篇毕业论文，以及业务活动图、功能模块图等配套文档。

## 附录F：导师可能提问与参考回答

**Q1：中试和本系统的关系是什么？**  
A：中试是科技成果放大验证的试验阶段；本系统不替代试验操作，而是管理广州生产力促进中心对外提供中试服务的全流程，包括申请、审核、派单、执行跟踪、复核和归档。

**Q2：为什么选广州生产力促进中心？**  
A：中心是市工信局平台介绍的科技服务机构，技术服务板块明确包含中试服务，有增城科研中试基地，职责与题目主题严格契合，不是通用系统套壳。

**Q3：Web 端和小程序端如何协同？**  
A：Web 是 **办公主渠道**——同一 Vue 前端按 **登录角色** 展示企业门户、技术人员门户或中心管理端；小程序是企业、技术人员的 **移动补充**。调度员、审核员仅在 Web 中心管理端操作。两端共用 Spring Boot 后端和 MySQL，通过 API 同步业务状态。

**Q4：业务活动图中的循环流程有什么实际意义？**  
A：材料不合规需补充重提、执行异常需重派、复核不合格需修改重交，都是真实业务中的回退机制；前端须能 **重新进入** 目标活动对应页面，保证流程可控、责任可追溯。

**Q5：泳道上方角色与下方活动是什么关系？这样搭配合理吗？**  
A：**合理且必须遵守**，与 `模板图/业务活动图.png` 一致：泳道标题是 **具体岗位**，括注 **（Web）** 或 **（Web/小程序）** 表示主要使用端；下方圆角矩形是 **该岗位在本流程中执行的操作**；跨泳道水平线表示 **岗位间业务交接**。活动须画在 **主要执行者** 所在列（企业「查看进度」画在企业列），便于按列汇总页面，见 §5.5、§5.10。

**Q6：5 个模块与业务活动图的关系？**  
A：功能模块图（5 二级、20 三级）已定稿；**5 张**业务活动图与二级模块 **一一对应**，由模块图反推，每图 ≥3 角色、**≥8 活动节点**（复杂模块 9—12），20 个三级功能全部落在核心活动节点上，见 §5.7 核对表。

**Q7：数据从哪里来？会不会拿不到政府内部数据？**  
A：开发测试使用模拟数据和业务流程填报数据即可，不依赖不可获取的政府内部核心数据，可行性已在 §7.4 说明。

**Q8：本科工作量能否完成？**  
A：技术栈成熟，无复杂算法；核心业务闭环清晰，按 16 周分阶段推进，先完成五条主流程再扩展统计等辅助功能，工作量可控。

**Q9：与现有科技成果转化平台有何区别？**  
A：现有平台多侧重成果展示和需求发布；本系统聚焦生产力促进中心的中试服务 **管理闭环**，强调审核—评估—派单—执行—复核—归档及 **Web 分角色 + 小程序移动补充**。
